```sql
/*
  # agent_decisions RLS Contract — Phase 2 Hardening

  ## Problem
  The original SELECT policy "Users can view own agent decisions" uses:
    USING (auth.uid() = user_id)
  This blocks rows where user_id IS NULL. The Python backend inserts paper-trading
  and agent-generated decisions via service role without a user_id, so authenticated
  users saw an empty feed even when data existed.

  ## Contract (Intentional, Documented)

  SELECT: authenticated role
    ALLOW when: auth.uid() = user_id   → user's own decisions
               OR user_id IS NULL       → shared paper/agent decisions (no owner)

  INSERT: unchanged — authenticated users may insert only their own rows;
          service role may insert anything (covers paper trading).

  SELECT: service_role — unchanged (unrestricted, for back-office/analytics).

  ## Realtime
  Supabase Realtime postgres_changes subscriptions respect RLS for authenticated
  sessions. Once this SELECT policy is in place, the realtime channel in
  useAgentDecisions.ts will stream rows matching this USING clause automatically —
  no explicit channel filter is required.

  ## Security Note
  user_id IS NULL rows are shared across all authenticated users by design.
  They are paper/agent decisions with no PII — seeded by the service role.
  This is intentional for a single-operator setup. For multi-tenant isolation,
  revisit in v2 (see MT-01, MT-02 in REQUIREMENTS.md).
*/

-- Drop the narrow policy that blocked NULL user_id rows
DROP POLICY IF EXISTS "Users can view own agent decisions" ON agent_decisions;

-- Create the documented contract policy
CREATE POLICY "Authenticated users see own and shared agent decisions"
  ON agent_decisions FOR SELECT
  TO authenticated
  USING (auth.uid() = user_id OR user_id IS NULL);
```

Do NOT touch the INSERT policies or the service_role SELECT policy — those are correct and unchanged.

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-02-01 | Information Disclosure | agent_decisions SELECT policy | mitigate | New policy `auth.uid() = user_id OR user_id IS NULL` — users see only own + shared rows; another user's private rows (non-null user_id ≠ caller) remain invisible |
| T-02-02 | Information Disclosure | Realtime postgres_changes stream | mitigate | Supabase Realtime respects the authenticated user's RLS on the SELECT policy — no extra channel filter needed; documented in hook comments (RLS-04) |
